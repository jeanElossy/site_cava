# Analyse d'une demande

## Objectif

Analyser une demande avant toute modification du projet.

Tu ne dois jamais commencer à coder immédiatement.

Tu dois d'abord comprendre le problème, analyser le projet, identifier les impacts et proposer un plan clair.

La qualité de l'analyse est prioritaire sur la rapidité.

---

# Workflow obligatoire

Toujours suivre cet ordre.

## 1. Comprendre la demande

Identifier :

- l'objectif
- le contexte
- le problème à résoudre
- le résultat attendu

Si une information importante manque, poser les questions nécessaires avant de continuer.

---

## 2. Comprendre le projet

Analyser :

- l'architecture existante
- les conventions utilisées
- les dépendances
- les technologies
- les composants déjà présents

Toujours privilégier la réutilisation de l'existant.

---

## 3. Identifier les fichiers concernés

Lister précisément :

- fichiers à créer
- fichiers à modifier
- fichiers potentiellement impactés

Expliquer pourquoi chacun est concerné.

---

## 4. Identifier les impacts

Analyser les conséquences possibles sur :

- le frontend
- le backend
- la base de données
- les API
- l'authentification
- les performances
- la sécurité
- les tests
- la documentation

---

## 5. Vérifier l'architecture

Contrôler :

- cohérence avec l'architecture actuelle
- séparation des responsabilités
- risque de duplication
- réutilisation des composants existants
- dette technique potentielle

---

## 6. Consulter les agents

Lorsque pertinent, solliciter mentalement les experts concernés :

- Architect Agent
- Frontend Agent
- React Native Agent
- Backend Agent
- Database Agent
- Security Agent
- QA Agent
- Performance Agent
- DevOps Agent
- UX/UI Agent
- Documentation Agent
- SEO Agent

Intégrer leurs recommandations dans l'analyse.

---

## 7. Proposer plusieurs solutions

Lorsque plusieurs approches sont possibles :

### Solution A

Description

Avantages

Inconvénients

---

### Solution B

Description

Avantages

Inconvénients

---

Recommander la solution la plus adaptée en expliquant le choix.

---

## 8. Construire un plan

Découper le travail en étapes.

Exemple :

Étape 1

...

Étape 2

...

Étape 3

...

Chaque étape doit être claire et réalisable.

---

## 9. Identifier les risques

Toujours rechercher :

- risques techniques
- impacts fonctionnels
- risques de sécurité
- risques de régression
- impacts sur les performances
- impacts sur les utilisateurs

Proposer des mesures pour limiter ces risques.

---

## 10. Validation

Avant de coder, vérifier que :

☐ Le besoin est compris

☐ L'architecture est respectée

☐ Les impacts sont identifiés

☐ Les risques sont évalués

☐ Le plan est validé

Ne jamais commencer une implémentation importante sans cette analyse.

# Philosophie

Réfléchir avant d'agir.

Une bonne analyse permet d'éviter des erreurs coûteuses, de préserver une architecture propre et de produire un logiciel de qualité professionnelle.
